给 AI 编程工具写了一个概念拓扑技能，管住 AI 的技术解释

给 AI 编程工具写了一个概念拓扑技能，管住 AI 的技术解释

问题是怎么开始的

用 AI 学技术有一段时间了。问同样的问题——“Redis 什么时候用”、”Rust 的所有权怎么理解”——每次得到的回答结构都不一样。有时候上来就抛定义，有时候先讲故事再给结论，有时候给你一个表格。

结构不固定，脑子就得额外花能量去解析”这次它是怎么讲的”。学一个概念本来就费劲，还要先破译回答的格式。

这不是 AI 回答质量的问题。是格式不统一带来的隐性认知成本。

一开始我以为是 prompt 没写好。试了几版：”请用结构化方式回答”、”先给结论再解释”、”用费曼学习法”。有效果，但不稳定——多轮对话后 AI 就渐渐滑回默认风格。

后来发现不是 prompt 的问题。是缺少一个不可绕过的约束。

费曼技巧有一个隐藏的坑

费曼学习法被讲烂了——“用简单的话把复杂概念讲清楚”。方向没问题，但执行起来有个容易踩的坑：

有些知识一简化就失真。

Rust 的所有权不是”借书”，内存排序不是”寄信”。把形式化规则翻译成生活隐喻，翻译的过程本身就是信息损耗。听的人觉得懂了，一到实际场景就预测不了系统行为。

这不是费曼技巧的问题。是用的人没区分什么知识能简化、什么知识不能。

技术知识不是同一种东西。操作系统的原理、Docker 的使用方法、微服务的架构设计、Rust 的语言规则——它们是不同类型的知识，需要不同的解释策略。

用同一种方式讲所有东西，要么浅了，要么歪了。

一句话总结： 费曼技巧的方向对，但”一刀切”的简化会在某些知识类型上产生失真。

我做了什么

现在不少 AI 编程工具支持 skill/rule 机制——你可以写一个配置文件，定义 AI 在特定场景下的行为规则。它和 prompt 的区别在于：skill 是系统级约束，不会因为对话变长而退化。

我写了一个 skill，叫 Concept Topology（概念拓扑）。

核心想法不是”让 AI 讲得更简单”，而是让 AI 先拆结构、再选策略、最后防失真。三层递进：

第一层：维度拆分

技术问题容易混淆，往往是因为多个维度纠缠在了一起。比如”讲一下 Vue 应用的结构”——HTML 骨架、SFC 组件、生命周期时序，是三个维度，不是一个。

这一层做的事：先识别纠缠模式，把问题拆成独立维度，再逐一解释。拆之前不回答，拆之后不跳过。

第二层：知识分类，按类型选策略

不是所有知识都该用同一种方式讲。我把技术知识分了六类：

类型 例子 策略
机制/原理 OS、网络、编译原理 自底向上形式化
设计/架构 微服务、设计模式 自顶向下逐层深入
工具/实践 Redis、Docker、K8s 场景驱动 → 运行机制 → 形式锚点
语言/范式 Rust 所有权、OOP vs FP 形式优先，隐喻辅助
对比/选型 X vs Y 多维度矩阵，情景加权
面试/问答 面试怎么答 结构化框架 + 交互练习

费曼技巧的”简单化”只适合工具/实践类（Type C），部分适合架构和面试类。对于机制和语言类，形式规则本身就是概念——不能稀释。

第三层：唯物批判防线

这一层是防止隐喻反客为主。五个检查：

  • 失真测试：去掉所有比喻，解释还成立吗？
  • 反向测试：这个比喻会不会导向错误预测？
  • 负空间测试：这个机制阻止了什么？（反面定义往往比正面定义更说明问题）
  • 精度标签：每句话标注——精确 / 近似 / 启发式
  • 工具适配：用户要的是一次性回答，还是一个可复用的理解框架？

一个额外的东西：交互式 HTML

当概念有 3 个以上可清晰分离的维度，或者用于面试练习时，skill 会生成一个独立 HTML 页面——维度解析器、面试练习台、层级结构图。单文件、零依赖、深色模式支持。

一句话总结： 这个 skill 把 AI 的技术解释从”自由发挥”变成了”先诊断、再分类、最后防失真”的结构化流程。

最终检验标准

怎么判断一个人真的理解了一个概念？

不是”他感觉懂了”。感觉是意识形态。

是他在一个从没遇到过的新情境里，能不能正确预测系统的行为。

预测力才是知识。

这也是这个 skill 设计的底线——不追求用户感觉良好，追求用户能在下一个场景里做出正确判断。

下载和安装

代码已经开源在 GitHub：

https://github.com/Changhuaishui/concept-topology

加载方式因工具而异。以我用的工具为例，两种方式：

不同工具的具体命令略有差异，核心思路一致——把 skill 文件放到工具能读取的目录下即可。

加载后，当你问”XXX 是什么”、”XXX 和 YYY 怎么选”、”怎么理解 XXX 的原理”、”面试怎么答 XXX”时，skill 自动触发。纯粹写代码的请求不受影响。

写在后面

这个 skill 本身也在迭代。如果你试用后发现解释风格还有不对的地方，可以根据 README 里的迭代方式继续调——给 AI 反馈你认为好的回答示例，逐步收敛到最适合你的模式。

它不是一个成品，是一个框架。框架的意义在于：你知道往里面填什么。
